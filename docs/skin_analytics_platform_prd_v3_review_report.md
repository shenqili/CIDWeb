# PRD v3 多维复审报告

## 1. 复审范围

复审对象：

- `D:\CIDWeb\docs\skin_analytics_platform_prd_v3.md`
- `D:\CIDWeb\docs\skin_analytics_spssau_visual_reference.md`

复审视角：

- 业务/功能完整性
- 架构可靠性
- 可用性/用户链路
- UI/组件规范
- QA/验收可执行性

说明：

- 本轮复审重点关注 `v3` 相比 `v2` 仍然残留的高优先级问题
- UI 残留问题主要结合上一轮 UI 评审结果与 `v3` 新增内容收敛

---

## 2. 本轮共识

`v3` 已经从“原则级 PRD”提升到“可设计级 PRD”。

相比 `v2`，本轮明确补进了：

- 品牌就是 `Brand`
- 一个项目可绑定多个品牌
- 普通用户只能在授权品牌范围内分别分析，不得跨品牌混合分析
- 超级管理员允许跨品牌、跨租户混合分析
- 主键映射与多源整合规则
- 派生变量回写并发布为新数据版本
- 问卷计分规则维护和项目级复制改写
- 高敏导出单人审批
- `MVP / Phase 1.5 / Phase 2` 分阶段设计

但 `v3` 仍未完全达到：

- 可直接实施
- 可直接验收
- 可直接运维

---

## 3. 剩余 P0 问题

### P0-1 受限分析空间技术边界仍不够具体

当前已定义超级管理员可做跨品牌、跨租户混合分析，但尚未明确：

- 是否独立计算池
- 是否独立数据副本区
- 是否允许导出明细
- 是否允许把产物回写到普通项目空间

建议：

- 在 `v3.1` 明确“受限分析空间”是独立执行域
- 默认仅允许输出聚合结果
- 禁止该空间的派生变量直接回写普通项目版本

### P0-2 模块验收仍不能直接转测试用例

`13.1` 已补充验收维度，但模块验收标准仍是摘要口径，无法直接用于系统级测试。

建议：

- 将 `13` 章改成模块级验收矩阵
- 至少拆出：
- 数据上传验收矩阵
- 主键映射验收矩阵
- 分析任务验收矩阵
- 派生变量发布验收矩阵
- 高敏导出审批验收矩阵

### P0-3 统计方法还缺黄金数据集与容差

当前已定义方法和自动推荐规则，但没有：

- 黄金数据集
- 主统计量期望值
- 数值容差
- 自动推荐分支测试基线

建议：

- 新增“统计方法验收附录”

---

## 4. 剩余 P1 问题

### P1-1 品牌切换状态机未定义

普通用户可在多品牌项目里切换品牌分别分析，但未定义切换后：

- 筛选规则是否清空
- 变量选择是否保留
- 模板是否重校验
- 未提交配置如何处理

建议：

- 在 `v3.1` 新增“品牌切换状态机”

### P1-2 首登路由规则仍不够具体

已有项目初始化链路，但没写清：

- 品牌配置管理员首登去哪
- 上传员首登去哪
- 分析师首登去哪

建议：

- 增加“角色首登路由规则”

### P1-3 快速分析 / 高级分析双模式还未正式结构化

当前工作台版式已补，但没有正式拆成产品模式。

建议：

- 快速分析：少量输入 + 系统推荐
- 高级分析：完整方法与参数

### P1-4 `ExecutionManifest` 还缺最小字段清单

当前只定义了对象，没有写死最小字段。

建议最小字段至少包含：

- `tenant_id`
- `project_id`
- `brand_config_version`
- `metric_catalog_version`
- `questionnaire_template_version`
- `dataset_version_hash`
- `analysis_template_version`
- `engine_version`
- `library_lock`
- `random_seed`
- `timezone`
- `export_template_version`

### P1-5 多品牌项目的版本绑定还不够原子化

当前仍未明确：

- 一个分析任务是绑定单品牌版本包，还是项目所有版本快照
- 版本切换是否成组发布

建议：

- 引入 `VersionBundle`
- 一个分析任务必须绑定一个品牌的一个版本包

### P1-6 任务状态机还缺部分成功与清理策略

建议补：

- `partial_success`
- 失败后中间产物清理
- 孤儿产物 TTL
- 队列公平调度和租户并发配额

### P1-7 高敏导出审批生命周期未完整定义

建议补：

- `pending_approval`
- `approved`
- `rejected`
- `expired`
- 审批有效期
- 下载次数限制
- 到期回收规则
- 默认禁止“申请人 = 审批人”

### P1-8 派生变量发布还缺测试化 lineage 规则

建议补：

- 新旧版本字段差异检查
- 重复发布行为
- 部分发布行为
- 冻结版与回滚后的行为

### P1-9 领域异常状态仍需补强

建议补的重点异常包括：

- 当前品牌下无可分析指标
- 模板版本升级导致旧模板失效
- `微生态编号` 缺失且 `RD` 冲突
- 派生变量来自旧版本而当前品牌不可用
- 高敏导出审批被拒绝或过期

### P1-10 UI 仍缺页面级设计决策

`v3` 已补分析型组件规范和工作台版式，但还缺：

- 可视化页面映射到你自己的页面块
- 图表工作台完整交互流
- 统计语义色
- 数据密度模式

建议：

- 单独补“分析工作台组件规范”
- 单独补“可视化页面映射表”

### P1-11 运维约束仍不够

建议补：

- `RPO / RTO`
- 备份频率
- 对象存储保留周期
- 中间文件保留期
- 导出包保留期
- 审计日志保留期

---

## 5. 建议的 v3.1 优先级

### 第一优先级

- 受限分析空间技术边界
- `ExecutionManifest` 最小字段规范
- 模块级验收矩阵
- 统计方法黄金数据集与容差

### 第二优先级

- 品牌切换状态机
- 快速分析 / 高级分析双模式
- `VersionBundle` 原子绑定规则
- 高敏导出审批生命周期

### 第三优先级

- 派生变量发布测试矩阵
- 领域异常状态矩阵
- UI 页面映射和组件细化
- 运维恢复目标与保留周期

---

## 6. 当前判断

当前 `v3` 已经足够进入：

- 业务评审
- 架构设计
- 数据模型设计
- 页面原型设计

但还不建议直接进入：

- 全量开发排期锁定
- 系统级测试设计冻结
- 生产运维方案冻结

在进入这些阶段前，建议先补一个 `v3.1`。
