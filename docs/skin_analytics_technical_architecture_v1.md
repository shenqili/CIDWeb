## 2. 总体架构原则

- 控制面共享，数据面隔离
- 配置驱动，版本绑定，结果可复现
- 原始数据不可覆盖，派生数据新版本发布
- 在线交互与科研导出分层
- 普通项目分析与平台级受限分析空间分离
- 先落可用 MVP，再逐步打开高级分析能力
- 当前阶段优先本地文件存储与 Docker 演示部署
- 通过存储抽象层和部署抽象层保留未来云端迁移能力

## 6. 核心数据流

### 6.1 项目初始化流

1. 创建项目
2. 绑定品牌
3. 为每个品牌绑定一个 `VersionBundle`
4. 设定导入模式和导出策略
5. 项目进入可用状态

### 6.2 数据导入流

1. 上传文件
2. 解析编码和字段
3. 识别主键和外部编号
4. 执行主键映射
5. 生成草稿数据版本
6. 预处理
7. 发布为正式数据版本

### 6.2.1 数据浏览流

1. 选择项目、品牌、数据集、数据版本
2. 后端按角色、品牌范围、数据版本状态过滤可见上下文
3. 加载字段元数据与可用筛选器
4. 加载首屏概览卡和分页明细
5. 用户调整字段筛选、样本筛选和排序
6. 后端重算样本量、分页结果和图表预览
7. 用户保存视图或推送到分析工作台

### 6.3 分析执行流

1. 选择品牌上下文
2. 绑定数据版本
3. 绑定 `VersionBundle`
4. 构建筛选规则
5. 提交分析任务
6. 生成 `ExecutionManifest`
7. 进入异步执行
8. 生成结果、图表、导出包

### 6.4 派生变量回写流

1. 分析成功
2. 生成 `DerivedVariableSet`
3. 用户选择发布
4. 基于旧版本生成新 `DatasetVersion`
5. 写入 lineage
6. 新版本进入后续分析

## 7. 主键与多源整合实现

### 7.1 主键策略

- 内部主键：`subject_id`
- 访问记录主键：`visit_id`
- 首选外部唯一编号：`微生态编号`
- 辅助映射编号：`RD`、`RD-问卷编号`

### 7.2 关键表

- `subject`
- `subject_identifier`
- `visit_record`
- `import_batch`
- `dataset_version`
- `dataset_version_lineage`

### 7.3 映射规则

- 有 `微生态编号` 优先自动匹配
- 无 `微生态编号` 时，`RD / RD-` 进入映射规则
- 弱特征不得自动合并
- 人工映射必须留痕

## 8. VersionBundle 设计

### 8.1 定义

`VersionBundle` 是分析任务绑定的最小原子配置单元：

- `brand_config_version`
- `metric_catalog_version`
- `questionnaire_template_version`

### 8.2 原则

- 一个分析任务只绑定一个品牌的一个 `VersionBundle`
- 项目可持有多个品牌的多个 `VersionBundle`
- 切换品牌等于切换 `VersionBundle`
- 历史结果永远绑定历史 `VersionBundle`

- 左：图种与图表列表
- 中：大图预览
- 右：属性面板

### 13.4 数据浏览图表预览

- 面向 `Data Explorer` 的图表预览只返回聚合结果
- 默认支持字段分布图、缺失率图、品牌对比图、版本对比图
- 图表预览应与当前筛选条件绑定，但不改变原始数据版本
- 当数据量较大时，优先使用缓存后的聚合结果或异步重算

## 14. 数据浏览查询约束

### 14.1 查询边界

- 只允许查询已发布 `DatasetVersion`
- 所有查询必须显式带上 `projectId`、`brandId`、`datasetVersionId`
- 品牌范围校验必须在后端执行，前端仅做体验层禁用

### 14.2 明细分页策略

- 明细表必须分页，不允许一次性返回整表
- 分页查询需支持排序、字段显示列和筛选条件
- 默认首屏返回概览卡 + 首 1 页数据，减少阻塞

### 14.3 筛选表达式

- 筛选规则采用结构化表达式，而不是拼接 SQL 文本
- 支持字段比较、区间、枚举、多选、缺失值过滤
- 推送到分析工作台时只传筛选定义，不复制明细数据
- Data Explorer 应生成可审计的 `querySnapshot`，供分页、图表预览和分析跳转复用

### 14.4 安全与性能

- 高敏明细字段默认不直接向普通角色暴露
- 跨品牌用户只返回授权品牌的结果集
- 常用概览统计和字段元数据可单独缓存
- 图表预览优先返回聚合数据，避免明细泄漏和大结果传输
- 明细表与图表预览必须绑定同一 `querySnapshot`，避免筛选口径漂移

## 16. 缓存与存储策略

### 15.1 缓存

- 只缓存可重建结果
- 缓存按租户命名空间隔离
- 高敏结果默认不进共享缓存

### 15.2 本地文件存储

当前阶段使用本地文件目录，建议按以下层级组织：

- `/storage/raw/{tenant}/{project}/{brand}/`
- `/storage/intermediate/{tenant}/{project}/{job}/`
- `/storage/exports/{tenant}/{project}/{job}/`
- `/storage/charts/{tenant}/{project}/{job}/`
- `/storage/manifests/{tenant}/{project}/{job}/`

保存内容：

- 原始文件
- 中间文件
- 导出包
- 图表产物
- `manifest.json`

### 15.3 未来对象存储迁移

未来切换到对象存储时，目录规则保持不变，仅底层实现替换为对象键前缀。

### 15.4 保留策略

- 数据版本保留策略
- 导出包保留策略
- 中间文件 TTL
- 审计日志保留策略

## 20. 未来云迁移路径

当前阶段不依赖云端即可完成完整演示。未来迁移时建议保持以下抽象不变：

- `StorageAdapter`：本地文件存储 -> 对象存储
- `QueueAdapter`：本地 Redis/Celery -> 云消息队列或托管 Redis
- `RuntimeAdapter`：Docker Compose -> 云端容器平台
- `ExportAdapter`：本地下载链接 -> 云端签名 URL

建议迁移顺序：

1. 先替换文件存储层
2. 再替换部署平台
3. 最后替换数据库、缓存、队列为托管服务
